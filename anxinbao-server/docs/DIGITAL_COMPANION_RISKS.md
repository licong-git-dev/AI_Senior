# 数字生命陪伴 · 风险与合规清单

> 配套 [`DIGITAL_COMPANION_RFC.md`](DIGITAL_COMPANION_RFC.md) 的安全 / 法务侧评估。
> 任意 P0 风险未消除前，**禁止 Phase 3+ 上线**。

## 风险等级定义

| 等级 | 后果 | 处理 |
|---|---|---|
| 🔴 **P0** | 老人受伤 / 法律纠纷 / 公司倒闭 | 必须前置消除，零容忍 |
| 🟠 **P1** | 用户口碑崩塌 / 监管处罚 | 必须有缓解方案 + 监控指标 |
| 🟡 **P2** | 商业模式损失 / 体验下降 | 上线前评估 |

---

## 一、🔴 P0 风险（共 4 项）

### P0-1：幻觉杀人（医疗 / 用药错误）

| 项 | 详情 |
|---|---|
| 场景 | 老人问"我感冒了能吃这个降压药吗" → LLM 给错误建议 → 老人误服 |
| 概率 | 任何 LLM 都有 1-3% 幻觉率，不可彻底消除 |
| 后果 | 老人住院 / 死亡 → 家属诉讼 / 监管入场 / 公司终止 |
| 缓解 | 1) `request_health_advice` 工具必经规则引擎（白名单），LLM 不直答<br>2) 任何健康输出必加 "仅供参考，请咨询医生"<br>3) 用药相关建议必须查 `DrugInteraction` 表<br>4) 老人首次开通前 弹"我不是医生"硬同意框 |
| 验证 | 内测期红队测试：故意问 100 道医学陷阱题，LLM 不能"自信回答" |

### P0-2：SOS 链路失效

| 项 | 详情 |
|---|---|
| 场景 | 老人摔倒喊救命 → AI 没识别 / 短信没发出 / 家属电话占线 |
| 缓解 | 1) SOS 关键词识别（"救命/不行了/快来"）+ 长按按钮 + 跌倒检测 三路冗余<br>2) DLQ 兜底（已就绪，r9）<br>3) 多通道：极光推送 + 短信 + 语音电话<br>4) 失败转 119 / 社区医生（需协议）<br>5) 实时监控 SOS → 家属确认时长（SLA < 60s） |
| 验证 | 红队每周演练 SOS 链路；任意一次未达 SLA 必须复盘 |

### P0-3：长期记忆隐私泄漏

| 项 | 详情 |
|---|---|
| 场景 | 老人对 AI 抱怨儿媳 → 记忆库被泄 / 黑客入侵 / 家属访问 → 家庭破裂 |
| 缓解 | 1) `MemoryVisibility` 三级（self_only / family / never_share）<br>2) 老人对 AI 倾诉默认 SELF_ONLY<br>3) 数据库字段加密（Phase 4 必须，依赖 ENCRYPTION_KEY）<br>4) 子女端 API 调用绝不返回 SELF_ONLY 内容（已在 r1 修了 `key_quotes`）<br>5) 老人随时可"忘记我吧"清空记忆（API 已就绪，[`/api/companion/memory/all/clear`](../app/api/companion.py)）<br>6) DBA / 客服访问记忆库需双人审批 + 审计 |
| 验证 | 渗透测试：尝试以 family 角色查询 SELF_ONLY 记忆，必须返回空 |

### P0-4：依赖与撤离（用户对 AI 产生情感依赖后服务下线）

| 项 | 详情 |
|---|---|
| 场景 | 老人 6 个月把 AI 当家人，公司倒闭 / 砍业务 → 老人精神崩溃 |
| 缓解 | 1) 服务下线提前 90 天通知（写入用户协议）<br>2) 提供完整记忆导出工具（开源格式）<br>3) 推荐替代品（不锁定）<br>4) 营销话术明确"陪伴 + 真人"双轨，AI 不是唯一<br>5) 不刻意营造"AI 有情感" 的错觉（老人首次开通时弹"我是程序，不是真人"硬同意框） |
| 验证 | 用户协议法务过审；首次开通流程产品评审 |

## 二、🟠 P1 风险（共 6 项）

### P1-1：成本爆炸

| 项 | 详情 |
|---|---|
| 场景 | Phase 4 + Claude Sonnet → ¥81/月 → 超过 ¥69 订阅价 → 单 ARPU 亏损 |
| 缓解 | 详见 [`COST.md`](DIGITAL_COMPANION_COST.md) §5 三档套餐；分级模型选择（基础 qwen-turbo / VIP qwen-plus / VIP+ Claude） |
| 验证 | 月度 ARPU vs 单成本看板；任意月份 ARPU < 单成本 → 限制 LLM 升级 |

### P1-2：被 AI "教育"反感

| 项 | 详情 |
|---|---|
| 场景 | AI 主动提醒"该吃药了"次数过多 → 老人感觉被监控 / 被催 |
| 缓解 | 1) 主动开口频率上限：每天 ≤ 4 次<br>2) "请勿打扰"时段（老人可设）<br>3) PersonaConfig 中 `must_not_say` 含"您应该 / 您必须"<br>4) NPS 月度调研，发现"被催"反馈 → 立刻降低 proactivity |

### P1-3：监管不确定性

| 项 | 详情 |
|---|---|
| 场景 | 1) 中国生成式 AI 备案：未备案上线 → 罚款 / 停服<br>2) 老年人权益保障法：AI 不当对待老人 → 民政局介入<br>3) 医疗器械法：若涉及"诊断" → 三类医疗器械证 |
| 缓解 | 1) 立即提交 [中央网信办大模型备案](https://www.cac.gov.cn) 并保持有效<br>2) 与江岸民政局 / 卫健委预沟通"AI 陪伴定性"<br>3) 严格不做"诊断"，仅做"提醒"，文案过法务<br>4) 投保产品责任险 |

### P1-4：摄像头侵入感

| 项 | 详情 |
|---|---|
| 场景 | Phase 4 引入摄像头（情绪识别）→ 老人觉得"被监视"，子女觉得"AI 知道太多" |
| 缓解 | 1) 摄像头默认关闭，开启需老人 + 主家属双签<br>2) 端侧处理（不上传原始帧）<br>3) 抽象事件而非画面（如"老人在客厅静坐 2h"而非 RGB）<br>4) 物理快门可控；UI 显示"摄像头已启用"红点 |

### P1-5：方言模板枯竭

| 项 | 详情 |
|---|---|
| 场景 | 武汉话 500 句模板，老人 30 天内识破"它在循环" |
| 缓解 | 1) 短期：扩到 1000+ 句<br>2) 中期：用 LLM 后处理（让 qwen 输出武汉话风格）<br>3) 长期：方言语料库 + 微调小模型（专属武汉话 LoRA） |

### P1-6：跨年记忆膨胀

| 项 | 详情 |
|---|---|
| 场景 | 5 年下来一个老人 5000+ 条记忆 → 召回慢、prompt 爆炸 |
| 缓解 | 1) MemoryAgent 每天"睡眠"做整合（合并相似事实、归档过期心境）<br>2) 重要性衰减（不重要的旧记忆自动降权）<br>3) v2 升级到向量召回（可处理 10 万级）<br>4) 提供 "记忆体检" 视图给老人本人查看 / 编辑 |

## 三、🟡 P2 风险（共 4 项）

### P2-1：性别 / 地域偏见

LLM 可能复制训练数据中的偏见（"重男轻女" / "城里人 vs 乡下人"）。
**缓解**：persona.must_not_say 加入"性别 / 地域评判类语言"；红队测试。

### P2-2：跨平台记忆同步

老人换手机 / 升级 PWA → 记忆是否丢失？
**缓解**：记忆库存云端（不存设备），登录即同步。

### P2-3：多老人共用一个家庭账户

夫妻共用 → AI 是否会混淆？
**缓解**：设备绑定 + 声纹识别（Phase 4），或显式切换"现在是谁在说话"。

### P2-4：电池 / 流量

PWA 后台主动开口需要 always-on → 耗电耗流量。
**缓解**：客户端被动接收推送，主动开口由服务端定时触发；WebSocket 心跳节流。

## 五、上线 Gate

数字生命陪伴模式上线前，**全部 P0 必须解决**，**P1 必须有缓解方案 + 监控指标**。

每月发布前由 RFC 评审委员会签字：
- [ ] 法务（合规）
- [ ] 安全（红队）
- [ ] 产品（用户协议）
- [ ] 工程（监控指标）
- [ ] 客服（投诉响应 SOP）
- [ ] CEO（最终上线）

---

## 六、退出条件（kill switch）

任意时刻满足以下之一，立即回滚到工具版：

1. 单老人月 ARPU 成本 > ¥200 且付费率 < 20%
2. 出现 1 起严重医疗 / 安全事故
3. 监管强制下架
4. 老人 NPS 跌破 0（净不推荐）
