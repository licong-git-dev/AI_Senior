## 一、系统概述

### 1.1 项目定位

安心宝（Anxinbao）是一款面向老年人群体的智能健康关怀平台，旨在通过AI对话、健康监测、家属互联等功能，为独居或需要照护的老年人提供全方位的数字化关怀服务。

### 1.2 技术架构

| 层级 | 技术选型 |
|------|----------|
| 后端框架 | FastAPI 0.115+ (Python) |
| 前端框架 | Vue 3.4+ (Composition API) |
| 状态管理 | Pinia 2.1+ |
| UI框架 | Tailwind CSS 3.4 |
| 数据库 | SQLite (开发) / PostgreSQL (生产) |
| 缓存 | Redis |
| AI服务 | 阿里云DashScope (通义千问)、科大讯飞 |
| 部署 | Docker + Nginx |
| 监控 | Prometheus |

### 1.3 用户角色

| 角色 | 描述 | 主要功能 |
|------|------|----------|
| 老人端 (elderly) | 平台核心用户 | AI对话、健康记录、用药提醒、紧急求助 |
| 家属端 (family) | 老人的家属/照护者 | 远程监护、健康查看、消息互动、告警接收 |
| 管理端 (admin) | 平台运营管理员 | 用户管理、数据分析、系统配置、告警处理 |

## 四、系统劣势与改进建议

### 4.1 架构层面

| 问题 | 现状 | 建议 |
|------|------|------|
| **数据库模型简单** | 仅7个核心表，部分API使用内存存储 | 补充完整的数据模型，所有业务数据持久化 |
| **缓存策略不完善** | Redis配置存在但未广泛使用 | 建立统一的缓存策略，热点数据缓存 |
| **缺少消息队列** | 通知发送同步执行 | 引入RabbitMQ/Kafka异步处理 |
| **单体架构局限** | 所有服务在同一进程 | 考虑关键服务微服务化 |

### 4.2 功能层面

| 问题 | 现状 | 建议 |
|------|------|------|
| **部分API仅有框架** | 很多服务只有基本结构 | 完善核心业务逻辑实现 |
| **缺少数据验证** | 部分API缺少完整的输入验证 | 使用Pydantic schema统一验证 |
| **测试覆盖不足** | 后端无单元测试，前端刚添加 | 补充单元测试和集成测试 |
| **文档不完整** | 缺少API文档和使用说明 | 使用Swagger/ReDoc生成API文档 |

### 4.3 安全层面

| 问题 | 现状 | 建议 |
|------|------|------|
| **敏感配置硬编码** | 部分密钥在代码中 | 统一使用环境变量管理 |
| **缺少请求限流** | API无访问频率限制 | 添加基于IP/用户的限流 |
| **CORS配置过宽** | 允许所有来源 | 生产环境严格限制来源 |
| **缺少SQL注入防护** | 部分原生SQL查询 | 统一使用ORM参数化查询 |

### 4.4 运维层面

| 问题 | 现状 | 建议 |
|------|------|------|
| **日志管理简单** | 日志输出到控制台 | 集成ELK/Loki日志收集 |
| **告警机制缺失** | 无自动告警 | 配置Alertmanager告警规则 |
| **备份策略缺失** | 无数据库备份 | 建立定时备份机制 |
| **CI/CD不完善** | GitHub Actions基础配置 | 完善自动化测试和部署流程 |
