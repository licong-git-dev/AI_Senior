# 任务17数据分析仪表盘 - 管理员配置指南

## 概述
本文档面向系统管理员，介绍如何配置和管理数据分析仪表盘系统。

## 1. 系统架构

### 1.1 技术栈
- **前端**: React 18 + TypeScript + TailwindCSS
- **图表库**: ECharts 6.0
- **后端**: Supabase (PostgreSQL + Edge Functions)
- **部署**: 静态网站托管

### 1.2 数据库表结构
| 表名 | 说明 | 关键字段 |
|------|------|----------|
| sensor_data | 传感器原始数据 | user_id, data_type, processed_value, timestamp |
| physiological_analysis | 生理数据分析 | user_id, heart_rate_variability, sleep_quality_score |
| behavior_patterns | 行为模式记录 | user_id, pattern_type, activity_trajectory |
| health_predictions | 健康预测数据 | user_id, prediction_type, predicted_values |
| health_alerts | 健康预警记录 | user_id, alert_type, severity, indicator_name |

## 2. 用户管理

### 2.1 添加新用户
**通过Supabase控制台**:
1. 登录Supabase控制台
2. 进入Authentication > Users
3. 点击"Invite user"或"Create user"
4. 填写用户信息（邮箱、密码）
5. 记录生成的user_id

**通过SQL**:
```sql
-- 创建用户（Supabase Auth会自动生成UUID）
INSERT INTO auth.users (email, encrypted_password, email_confirmed_at)
VALUES ('user@example.com', crypt('password', gen_salt('bf')), NOW());
```

### 2.2 角色权限配置
当前支持的角色：elderly, family, doctor, nurse, admin

**配置用户角色**（需要创建profiles表）:
```sql
-- 创建用户配置表
CREATE TABLE IF NOT EXISTS profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id),
  role VARCHAR(20) NOT NULL DEFAULT 'elderly',
  display_name VARCHAR(100),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 设置用户角色
UPDATE profiles 
SET role = 'doctor' 
WHERE id = '用户UUID';
```

### 2.3 删除用户

## 3. 预警阈值配置

### 3.1 当前预警规则
health_alerts表中的预警通过以下规则生成：

| 指标 | 正常范围 | 预警条件 |
|------|----------|----------|
| 心率 | 60-100 bpm | <50 或 >120 |
| 收缩压 | 90-140 mmHg | <90 或 >160 |
| 舒张压 | 60-90 mmHg | <60 或 >100 |
| 血糖 | 3.9-6.1 mmol/L | <3.5 或 >8.0 |
| 体温 | 36.1-37.2°C | <35.5 或 >38.0 |

### 3.2 修改预警阈值
**方法1: 通过Edge Function配置**
编辑Edge Function配置文件添加阈值参数：
```typescript
const ALERT_THRESHOLDS = {
  heart_rate: { min: 60, max: 100 },
  blood_pressure_systolic: { min: 90, max: 140 },
  blood_pressure_diastolic: { min: 60, max: 90 },
  blood_glucose: { min: 3.9, max: 6.1 },
  temperature: { min: 36.1, max: 37.2 }
}
```

**方法2: 创建配置表**
```sql
CREATE TABLE alert_thresholds (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  indicator_name VARCHAR(100) NOT NULL,
  min_value DECIMAL(10,2),
  max_value DECIMAL(10,2),
  severity VARCHAR(20) DEFAULT 'medium',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 设置个性化阈值
INSERT INTO alert_thresholds (user_id, indicator_name, min_value, max_value, severity)
VALUES ('用户UUID', '心率', 55, 110, 'medium');
```

### 3.3 预警严重程度规则
```typescript
// 根据偏离程度判断严重程度
function calculateSeverity(value, min, max) {
  if (value < min) {
    const deviation = (min - value) / min
    if (deviation > 0.3) return 'high'      // 偏离30%以上
    if (deviation > 0.15) return 'medium'   // 偏离15-30%
    return 'low'                            // 偏离15%以内
  }
  if (value > max) {
    const deviation = (value - max) / max
    if (deviation > 0.3) return 'high'
    if (deviation > 0.15) return 'medium'
    return 'low'
  }
  return 'normal'
}
```

## 4. 数据管理

### 4.1 数据清理策略
**自动清理旧数据**（建议配置定时任务）:
```sql
-- 清理90天前的传感器数据
DELETE FROM sensor_data 
WHERE timestamp < NOW() - INTERVAL '90 days';

-- 清理已解决的旧预警
DELETE FROM health_alerts 
WHERE resolved = true 
AND resolved_at < NOW() - INTERVAL '30 days';

-- 清理过期的预测数据
DELETE FROM health_predictions 
WHERE prediction_time < NOW() - INTERVAL '60 days';
```

**创建定时清理任务**（使用pg_cron）:
```sql
-- 每天凌晨2点执行清理
SELECT cron.schedule(
  'cleanup-old-data',
  '0 2 * * *',
  $$
  DELETE FROM sensor_data WHERE timestamp < NOW() - INTERVAL '90 days';
  DELETE FROM health_alerts WHERE resolved = true AND resolved_at < NOW() - INTERVAL '30 days';
  $$
);
```

### 4.2 数据备份
**手动备份**:
```bash
# 使用pg_dump备份数据库
pg_dump -h db.bmaarkhvsuqsnvvbtcsa.supabase.co \
  -U postgres \
  -d postgres \
  --data-only \
  -t sensor_data \
  -t physiological_analysis \
  -t behavior_patterns \
  -t health_predictions \
  -t health_alerts \
  > backup_$(date +%Y%m%d).sql
```

**自动备份策略**:
- Supabase自动每日备份

- 可在Supabase控制台恢复

### 4.3 数据导出
**批量导出用户数据**:

## 5. 性能优化

### 5.1 数据库索引
**必要索引**（应该已创建）:
```sql
-- 传感器数据索引
CREATE INDEX idx_sensor_user_time ON sensor_data(user_id, timestamp DESC);
CREATE INDEX idx_sensor_type ON sensor_data(data_type);

-- 预警索引
CREATE INDEX idx_alerts_user_time ON health_alerts(user_id, alert_time DESC);
CREATE INDEX idx_alerts_resolved ON health_alerts(resolved, alert_time DESC);

-- 分析数据索引
CREATE INDEX idx_physio_user_time ON physiological_analysis(user_id, analysis_time DESC);
CREATE INDEX idx_behavior_user_time ON behavior_patterns(user_id, detection_time DESC);
```

**检查索引使用情况**:
```sql
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan DESC;
```

### 5.2 查询优化
**避免全表扫描**:
```sql
-- 不好的查询
SELECT * FROM sensor_data;

-- 优化后的查询
SELECT * FROM sensor_data 
WHERE user_id = '用户UUID' 
AND timestamp >= NOW() - INTERVAL '7 days'
LIMIT 100;
```

**使用EXPLAIN分析查询**:
```sql
EXPLAIN ANALYZE
SELECT * FROM sensor_data 
WHERE user_id = '用户UUID' 
AND timestamp >= NOW() - INTERVAL '7 days';
```

### 5.3 缓存策略
前端已实现：
- 数据缓存：100条最近记录
- 自动刷新：每60秒
- 实时订阅：可选开启

## 7. 安全配置

### 7.1 Row Level Security (RLS)
**确保RLS策略已启用**:
```sql
-- 检查RLS状态
SELECT tablename, rowsecurity 
FROM pg_tables 
WHERE schemaname = 'public';

-- 启用RLS
ALTER TABLE sensor_data ENABLE ROW LEVEL SECURITY;
ALTER TABLE health_alerts ENABLE ROW LEVEL SECURITY;

-- 创建访问策略
CREATE POLICY "Users can view own data"
ON sensor_data FOR SELECT
USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own data"
ON sensor_data FOR INSERT
WITH CHECK (auth.uid() = user_id);
```

### 7.2 API密钥管理
**密钥类型**:
- anon key: 前端使用，仅限RLS保护的操作
- service_role key: 后端使用，跳过RLS

**密钥轮换**:
1. 在Supabase控制台生成新密钥
2. 更新前端环境变量
3. 重新部署应用
4. 撤销旧密钥

### 7.3 数据加密
- 传输加密：强制HTTPS
- 存储加密：Supabase自动加密
- 敏感字段：考虑应用层加密

## 8. 故障排查

### 8.1 常见问题

#### 问题1: 图表不显示数据
**排查步骤**:
1. 检查浏览器控制台是否有错误
2. 验证数据库中是否有数据
```sql
SELECT COUNT(*) FROM sensor_data WHERE user_id = '用户UUID';
```
3. 检查时间范围筛选是否合理
4. 验证RLS策略是否正确

#### 问题2: 实时更新不工作
**排查步骤**:
1. 检查Supabase Realtime是否启用
2. 验证网络连接
3. 检查浏览器控制台WebSocket连接
4. 重启Realtime服务

#### 问题3: 预警不生成
**排查步骤**:
1. 检查health_alerts表是否有写入权限
2. 验证Edge Function是否正常运行
3. 检查预警阈值配置
4. 查看Edge Function日志

### 8.2 性能问题
**症状**: 页面加载慢
**解决方法**:
1. 检查数据库索引
2. 优化查询（添加LIMIT）
3. 启用前端缓存
4. 减少时间范围

**症状**: 图表渲染慢
**解决方法**:
1. 减少数据点数量
2. 使用数据采样
3. 启用懒加载
